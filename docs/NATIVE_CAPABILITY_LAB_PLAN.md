# 火山原生能力实验线方案

最后更新：2026-04-17

## 1. 方案目的

这份文档只做一件事：

- 为当前酒店展厅项目设计一条**独立的火山原生能力实验线**

目标不是替换当前项目，也不是立即重构主链，而是单独验证下面几项能力在火山原生链里到底能不能稳定成立：

- 原生实时闲聊
- 官方联网搜索
- 原生视觉理解
- 官方知识库 / MCP
- 原生字幕与文本结果回流

这条实验线必须和当前项目隔离，避免影响现有 `Hotel-current` 的联调和线上验证。

## 2. 当前主项目现状

当前主项目已经不是一个单纯的官方 Demo，而是一套经过业务编排的系统：

- 火山 RTC / VoiceChat 负责实时媒体和原生对话壳
- 酒店事实问题由后端接管，并通过 FastGPT 检索
- 收益分析、调价方案、飞书推送由后端收益 MCP 接管
- 前端已经有舞台页、字幕显示、打断闸门和多链路所有权控制

因此，如果继续在当前项目里混着测火山原生能力，会分不清以下问题：

- 是火山原生能力本身不稳定
- 还是后端接管逻辑把原生链盖掉了
- 还是字幕/回调/前端展示链条本身有问题

所以需要单独拉一条实验线。

## 4. 火山原生能力边界判断

基于官方文档，目前可以先做出这些判断：

### 4.1 官方明确支持的

- 实时语音对话
- 智能打断
- 实时字幕
- 联网搜索
- 视觉理解
- 官方知识库
- MCP / Remote MCP
- Function Calling

### 4.2 官方支持但不代表可直接接本地服务

以下能力虽然能做，但通常要求：

- 服务有公网可访问地址
- 或者通过云端 MCP / Remote MCP 暴露

包括：

- 本地 FastGPT
- 本地知识库
- 本地 MCP
- 本地私有工具服务

也就是说，火山原生链更适合直接接：

- 官方联网搜索
- 官方知识库
- 官方 MCP

而不是直接吃你 Mac 上的 `localhost` 工具。

## 5. 实验线目标

实验线只验证一件事：

**火山原生链能否在不依赖当前主项目后端接管的前提下，稳定跑通关键能力。**

需要逐项回答：

1. 原生闲聊能不能稳定回流文本
2. 官方联网搜索能不能稳定出结果
3. 原生视觉能不能真正看见并回答
4. 官方知识库 / MCP 能不能真的被触发
5. 客户端字幕与服务端回调能不能构成稳定闭环

## 6. 实验线设计原则

- 不改当前 `Hotel-current` 主项目
- 不占用当前主服务的逻辑入口
- 不复用当前复杂后端编排
- 不接入当前 FastGPT 主链
- 不接入当前收益调价主链
- 不接入当前复杂舞台字幕策略
- 只做“最小能力验证闭环”

## 7. 实验线建议目录

建议新建独立目录，例如：

```text
/Users/openclaw/hotel-full-package/Hotel-native-lab
```

或者：

```text
/Users/openclaw/hotel-full-package/rtc-native-capability-lab
```

这个目录只做实验，不承载当前业务。

## 8. 实验线最小组成

### 8.1 最小后端

只保留这些能力：

- `bootstrap`
- `start / stop voicechat`
- 基本 RTC 会话管理
- 回调接收
- 字幕事件落日志
- 会话状态查看

不接这些：

- FastGPT
- 收益 MCP
- 后端酒店知识接管
- 后端视觉兜底
- 主项目里的复杂打断编排

### 8.2 最小前端

只保留这些能力：

- 进房
- 开麦克风
- 开摄像头
- 发一轮文本或语音
- 展示原始 RTC 字幕
- 展示服务端回调日志

不做这些：

- 舞台页包装
- 复杂动画
- 数字人展示 UI
- 多层字幕美化

## 9. 分阶段验证计划

### 阶段 1：验证原生闲聊闭环

目标：

- 只测普通原生对话

测试问题建议：

- 你叫什么名字
- 你会什么
- 你平时会无聊吗

验收标准：

- 原生链能回答
- 客户端能收到 AI 字幕
- 回调可观测到轮次变化
- 文本结果能稳定被记录

如果这一阶段都不稳定，后面的视觉、联网、知识库都先不继续。

### 阶段 2：验证官方联网搜索

目标：

- 只开官方 Web Search

测试问题建议：

- 合肥今天天气怎么样
- 附近有没有商场
- 从高铁站过来要多久

验收标准：

- 能触发联网链
- 回答内容与问题相关
- 字幕文本能回前端
- 不把酒店事实问题误走联网搜索

### 阶段 3：验证原生视觉理解

目标：

- 只开官方视觉，不接后端视觉兜底

测试问题建议：

- 你看看我手里拿的是什么
- 画面里有什么
- 你能看到我举着一张纸吗

验收标准：

- 视觉输入可以上行
- 至少能回答简单观察问题
- 客户端字幕能回流
- 结果与画面内容相关

### 阶段 4：验证官方知识库 / MCP

目标：

- 不先接本地 FastGPT
- 先接火山官方知识库与 MCP

验证方式：

- 使用 Viking Knowledge
- 创建官方 Knowledge MCP Server
- 在 `LLMConfig.MCP` 中只配置必要工具

验收标准：

- 可以触发 `search_knowledge`
- 回答来自知识库
- 文本结果能回流

## 10. 实验线和主项目的边界

### 10.1 实验线里适合验证的

- 闲聊
- 官方联网搜索
- 原生视觉观察
- 官方知识库
- 官方 MCP
- 文本回流稳定性

### 10.2 暂时不迁过去的

- 酒店知识主问答
- FastGPT 本地知识库
- 收益调价链
- 飞书推送
- 执行确认
- 复杂多链路打断编排
- 当前舞台页字幕展示体系

## 11. 为什么不能直接在当前项目里试

当前项目已经叠加了大量业务控制：

- 单轮单所有者
- 后端接管播报
- 字幕优先级控制
- FastGPT 检索
- 收益 MCP
- 前端抢断与 preempt 逻辑

如果继续在这个项目里直接验证火山原生能力，会很容易出现：

- 测出来的不是“原生能力”
- 而是“原生能力 + 当前业务编排”的混合结果

实验线的价值，就是把这件事拆开。

## 12. 最终输出物

实验线完成后，需要输出一张明确结论表：

| 能力 | 是否可用 | 是否稳定 | 是否建议迁入主链 |
|---|---|---|---|
| 原生闲聊 | 待验证 | 待验证 | 待判断 |
| 官方联网搜索 | 待验证 | 待验证 | 待判断 |
| 原生视觉理解 | 待验证 | 待验证 | 待判断 |
| 官方知识库 | 待验证 | 待验证 | 待判断 |
| 官方 MCP | 待验证 | 待验证 | 待判断 |
| 原生文本回流 | 待验证 | 待验证 | 待判断 |

## 13. 当前建议结论

在实验线跑完之前，主项目仍然建议保持：

- 酒店事实问题继续走后端 FastGPT
- 收益问题继续走后端 MCP
- 原生链继续负责闲聊、迎宾和轻引导

也就是说：

**先把火山原生能力独立验清楚，再决定哪些值得迁过去。**
